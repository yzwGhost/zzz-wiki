## Context

现有统一同步入口已经能返回 `summary.subtasks` 和失败项摘要，因此这轮不需要重写同步主流程，只需要围绕“失败子任务”增加单独重试调用链。最小策略是：仅支持人工触发单次重试，仅面向角色、音擎、驱动盘三个真实同步子任务，并把重试动作额外写入日志。

## Goals / Non-Goals

**Goals:**
- 提供失败子任务的单次重试 API
- 在同步中心中直接展示和触发失败项重试
- 记录可区分原始失败与重试结果的日志

**Non-Goals:**
- 不实现自动重试
- 不实现定时同步
- 不实现增量同步
- 不改写现有 crawler adapter 或 `sync_catalog` 主流程顺序

## Decisions

- 重试入口单独建模，而不是复用 `runTask` 的宽泛语义。
  - 这样可以在类型、日志和 UI 上明确标识“这是重试动作”。
- 重试仅针对真实三类子任务：`fetch_mhy_agents`、`fetch_mhy_weapons`、`fetch_mhy_drive_discs`。
  - 这样范围最小，也符合当前统一同步的组成。
- 重试动作额外写入一条带重试元数据的 `sync_logs` 记录。
  - 这样不会破坏子任务原始日志，同时能追踪“是否进行了重试”和“重试是否成功”。
- 同步中心优先基于最近一次 `sync_catalog` 日志里的失败项展示重试按钮。
  - 这样避免引入复杂历史关联或队列查询。

## Risks / Trade-offs

- [Risk] 最近一次日志不是 `sync_catalog` 时，页面无失败项可重试 → Mitigation: 明确当前只针对最近一次统一同步失败项提供重试
- [Risk] 重试成功后原始失败日志仍存在，可能让用户困惑 → Mitigation: 额外重试日志明确标记来源任务、来源日志和重试结果
