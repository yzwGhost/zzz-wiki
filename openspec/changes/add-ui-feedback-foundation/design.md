## Context

当前项目的 React 页面已经通过统一 service 获取 SQLite / Electron 数据，但用户可见反馈仍分散在各个页面和按钮内。角色列表、角色详情、首页和同步中心都存在异步加载或环境依赖，但没有统一的加载、成功、失败规范；同步中心虽然有日志结构，但前端缺少稳定的运行中和失败提示。第一批改造需要覆盖多个页面模块，因此应采用低风险的共享反馈层，而不是逐页各写一套临时逻辑。

## Goals / Non-Goals

**Goals:**
- 统一异步交互状态模型，约定 `idle / loading / success / error`。
- 为首页、角色列表、角色详情、音擎列表、驱动盘列表和同步中心补齐页面级加载态、空态和失败态。
- 为手动同步、收藏及其他已有异步按钮补齐 loading、防连点和结果提示。
- 对 Electron bridge 不可用、Python 同步失败、数据库读取失败等错误做前端文案收口。

**Non-Goals:**
- 不实现复杂任务进度条、分阶段实时流式日志或自动轮询调度。
- 不引入 Redux、React Query 等新的重型状态方案。
- 不重写现有页面结构，只在当前 feature / page 组织上增强反馈能力。

## Decisions

### 1. 以轻量共享工具 + 页面内状态为主，不新增全局复杂框架
优先新增渲染层共享反馈工具和少量通用组件，继续复用现有 store 和 service。页面级加载与错误优先在页面内表达，轻成功提示使用 Ant Design `message`，任务级反馈使用已有同步中心信息区。

备选方案：
- 全局事件总线：不选，会让状态来源更分散。
- 直接每页各自实现：不选，后续难以统一维护。

### 2. 页面级反馈优先使用 Skeleton / Result / Empty，而不是单一全局 Spinner
资料站页面需要保持结构感。列表和详情页应保留布局骨架，只替换内容区状态；同步中心和环境错误则使用更明确的状态面板。

备选方案：
- 全局遮罩加载：不选，会影响桌面资料浏览体验。
- 所有场景只用 `Spin`：不选，可读性不足。

### 3. 异步按钮统一收口到受控状态与标准提示
所有会触发异步行为的按钮需要具备 `loading`、防重复点击和明确的成功/失败反馈。结果提示统一通过共享错误格式化工具和轻提示 API 输出，避免各组件直接拼接技术错误。

备选方案：
- 保持按钮只改文案不加 loading：不选，用户无法判断任务是否执行。

### 4. 错误归一化先在前端完成，不调整 Electron 协议
本轮不改大面积 IPC 协议，而是在渲染层增加错误规范化函数，将已有异常、bridge 缺失和 service 抛错转成统一展示文本。这能在不扩大变更面的情况下先收敛用户可见反馈。

## Risks / Trade-offs

- [多个页面分别接入加载态后样式不一致] -> 通过共享 Skeleton 区块和统一空/错态文案减少分叉。
- [轻量实现导致任务级状态信息不够丰富] -> 本轮只解决“可见、可重试、可理解”，更细进度放到后续批次。
- [错误来源复杂，文案归一化覆盖不全] -> 先覆盖 bridge、同步、读取三类高频错误，保留默认兜底文案。

## Migration Plan

1. 增加共享反馈状态与错误格式化工具。
2. 补齐按钮、首页、角色列表、角色详情、音擎列表、驱动盘列表、同步中心的第一批反馈。
3. 校验页面类型和同步中心行为，确保 Electron 环境内外都有明确提示。
4. 通过 OpenSpec、web/desktop typecheck 验证变更。
