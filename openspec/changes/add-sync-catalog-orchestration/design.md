## Context

当前桌面端同步模型是“一个前端按钮对应一个 Python 任务”，角色、音擎、驱动盘各自分散执行。已有子任务链路已经可用，因此本轮不需要重写 crawler 任务，只需要在桌面编排层增加顺序执行与结果汇总。

## Goals / Non-Goals

**Goals:**
- 提供单一 `sync_catalog` 入口
- 顺序串联角色、音擎、驱动盘同步
- 为统一入口记录聚合日志并返回稳定汇总结构
- 让同步中心默认围绕统一入口工作

**Non-Goals:**
- 不实现失败重试
- 不实现自动定时同步
- 不实现增量同步
- 不重构现有 adapter、cleaner、exporter 结构

## Decisions

- 在 Electron 主进程新增轻量编排服务，而不是在 crawler 中新建复杂任务树。
  - 这样可以直接复用现有 Python 单任务入口，并最小化对 crawler 结构的影响。
- 统一任务的子任务顺序固定为角色、音擎、驱动盘。
  - 这样结果稳定，便于同步中心展示。
- 聚合日志单独写入 `sync_logs`，同时保留子任务各自已有日志。
  - 这样既保留统一视角，也不破坏现有单任务可追踪性。
- 共享类型中新增统一同步摘要结构，并允许成功/失败结果都携带摘要。
  - 这样前端和日志展示层可以统一消费。

## Risks / Trade-offs

- [Risk] 统一任务与子任务各自写日志，会让日志数量增多 → Mitigation: 同步中心优先展示统一入口并通过任务名区分聚合日志与子任务日志
- [Risk] 某个子任务失败时前面已成功的子任务不会回滚 → Mitigation: 本轮明确只做最小编排，不做事务回滚或重试
