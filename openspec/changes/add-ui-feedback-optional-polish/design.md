## Context

当前渲染层已经有页面内的空态、错态和同步状态提示，但还缺两个面向异常与诊断的增强层。第一，全局渲染错误没有统一兜底，极端情况下用户仍可能看到不可恢复的空白页。第二，同步中心虽然有任务切换和日志列表，但对于“任务正在什么阶段”“最近一次日志里还有什么细节”仍然只能靠简短摘要判断。第三批适合作为可选增强，在不改变桥接协议主干的前提下，把这些体验补齐。

## Goals / Non-Goals

**Goals:**
- 增加全局错误边界或路由级错误兜底页。
- 增加同步阶段提示，但只展示真实可推断的阶段，不伪造进度百分比。
- 让同步日志可以展开查看更多上下文，如错误码、输出路径、stdout/stderr 摘要等。

**Non-Goals:**
- 不实现真正的流式日志或后端推送进度。
- 不做自动错误上报服务。
- 不构建完整的任务队列或后台任务管理器。

## Decisions

### 1. 使用 React 错误边界和路由错误页双层兜底
根渲染层用自定义错误边界拦截渲染异常，同时在 Router 层保留错误页入口，确保组件异常和路由异常都能落到可读的错误界面。

### 2. 同步阶段只基于当前可观测状态推断
阶段仅区分 `idle / preparing / running / success / failed`，其中 `preparing` 和 `running` 由前端按钮状态与任务返回结果推导，不引入虚假的百分比或分步日志。

### 3. 同步日志详情以可选字段扩展现有 Summary 模型
不另开全新日志详情接口，先在现有 `SyncLogSummary` 上增加可选 detail 字段，Electron 侧从 `payload_json` 中做最小解析，前端表格使用 expandable row 展示。

## Risks / Trade-offs

- [错误边界可能掩盖真实调试上下文] -> 页面中保留错误摘要和刷新/返回入口，不吞掉控制台错误。
- [阶段反馈粒度有限] -> 明确采用“真实可推断阶段”，不把它包装成完整进度系统。
- [日志详情字段不同步不齐全] -> 所有 detail 字段保持可选，缺失时显示占位文案。

## Migration Plan

1. 增加全局错误边界与错误页组件。
2. 扩展同步日志 summary 字段并解析 payload。
3. 在同步中心接入阶段条与可展开日志详情。
4. 做 typecheck、build 和 OpenSpec 验证。

## Open Questions

- 后续若需要完整日志页，是否拆成单独路由而不是继续堆在同步中心内。
- 第四批若接入真正的进度事件，是否复用当前阶段组件还是升级成时间线组件。
