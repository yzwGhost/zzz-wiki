## Context

当前项目已经有第一批反馈基础设施，包括 skeleton、空/错态面板和同步执行结果提示。第二批不再解决“有没有反馈”，而是解决“反馈是否足够顺手、是否像一个完整产品”。范围主要集中在 React 渲染层，不改 Electron / Python 接口协议，也不引入新的状态管理库。

## Goals / Non-Goals

**Goals:**
- 增强收藏按钮的视觉状态与提示一致性。
- 让角色、音擎、驱动盘列表页更明确地表达已应用筛选条件和结果数量变化。
- 为页面容器、卡片列表和局部内容区增加轻量过渡动效。
- 让同步中心更清晰地呈现“最近一次同步”“当前动作结果”“日志状态标签”和预留重试位。

**Non-Goals:**
- 不做长任务实时进度流。
- 不做自动重试、轮询刷新或定时同步。
- 不做大型动画系统或复杂页面重构。

## Decisions

### 1. 继续复用现有反馈基础，不新增第二套通知或状态系统
第二批直接建立在第一批的 `feedback.ts`、`FeedbackStatePanel` 和统一样式上，避免并行两套交互规范。

### 2. 搜索/筛选反馈优先通过“结果工具条”表达，而不是把提示塞进筛选控件内部
筛选控件保持简洁，把“当前结果数、已应用条件、清空入口”收敛到列表上方的单独结果工具条，更适合资料站结构，也便于三类列表页复用。

### 3. 过渡动效只做轻量级 CSS 进入动画
本轮不引入动画库，使用统一的淡入/上移动效和状态变换，保证性能和维护成本可控。

### 4. 同步中心增强以信息结构为主，不伪造进度
同步中心强化状态卡、日志标签和任务摘要，但不假装拥有真实分阶段进度，避免误导用户。

## Risks / Trade-offs

- [结果工具条信息过多导致页面拥挤] -> 仅展示核心信息：结果数、已应用条件、清空入口。
- [动效过多影响资料浏览] -> 动效持续时间控制在 160ms - 220ms，避免夸张位移。
- [收藏提示频繁打扰用户] -> 收藏仍用轻提示，但按钮本身增加明显状态变化，减少对全局消息的依赖。

## Migration Plan

1. 增加结果工具条和轻量通知收口。
2. 接入三类列表页和收藏按钮。
3. 补 shell/content/card 进入动效与状态样式。
4. 强化同步中心状态卡和日志区块。
